# Release Notes — Version 5.1.0 (Production Hardening)

**Version:** 5.1.0  
**Date:** 2026-07-03  
**Tag:** `v5.1.0`

## Summary

Version 5.1.0 is the **Production Hardening** release. It extends the pilot-ready 5.0+ product with organization API key authentication on reporting, staff TOTP and OIDC enrollment, Stripe billing scaffold, retention enforcement jobs, portal push notification scaffold, and PostgreSQL materialized reporting views — all behind feature flags and documented **Partial** scope limits.

Six of eight 5.1 epics ship as **Partial ✅**; **production SMS** and **LLM document summary UI** are explicitly deferred to **5.2+**. See [`docs/governance/version-5.1-scope.md`](../governance/version-5.1-scope.md).

## Shipped in v5.1.0

## API key authentication

Machine clients can authenticate reporting operations when `ENABLE_ENTERPRISE=true`:

| Capability | Details                                                       |
| ---------- | ------------------------------------------------------------- |
| Endpoint   | `GET /reporting/operations`                                   |
| Auth       | `X-API-Key: vrd_live_…` or `Authorization: Bearer vrd_live_…` |
| Scopes     | `read` required; `last_used_at` updated on success            |
| Staff JWT  | `read_only` role remains supported on all reporting routes    |

Per-route rate limiting UI, key rotation automation, and public developer portal → **5.2+**.

## Identity enrollment

Staff enrollment beyond the v5.0 SSO/MFA readiness scaffold:

| Capability | Endpoints                                                            |
| ---------- | -------------------------------------------------------------------- |
| TOTP       | `POST /enterprise/mfa/totp/enroll`, `confirm`, `GET`, `DELETE`       |
| OIDC link  | `POST /enterprise/sso/enrollment/start`, `callback`, `GET`, `DELETE` |
| Gate       | `ENABLE_ENTERPRISE=true`                                             |

SCIM provisioning, multi-IdP federation, and passwordless → **5.2+**.

## Billing

Stripe subscription scaffold behind `ENABLE_BILLING`:

| Endpoint                        | Description                                         |
| ------------------------------- | --------------------------------------------------- |
| `GET /billing/status`           | Readiness and blockers                              |
| `POST /billing/setup`           | Create Stripe customer (admin)                      |
| `POST /billing/subscribe`       | Create subscription (admin)                         |
| `POST /billing/webhooks/stripe` | Webhook handler (signature verified)                |
| Org summary                     | `billing` embedded on `GET /org-admin/organization` |

Usage-based metering, invoicing PDFs, and dunning automation → **5.2+**.

## Compliance enforcement

Retention policy enforcement behind `ENABLE_COMPLIANCE_ENFORCEMENT`:

| Capability | Details                                                          |
| ---------- | ---------------------------------------------------------------- |
| Status     | `GET /compliance/enforcement/status`                             |
| Audit log  | `GET /compliance/enforcement/runs`                               |
| Manual run | `POST /compliance/enforcement/run` (admin)                       |
| Worker     | `retention_enforcement_scan` at `0 3 * * *` UTC                  |
| Scopes     | Soft-delete for `documents`, `communications`, `client_profiles` |
| Deferred   | `audit_logs` scope records skipped runs (append-only)            |

## Portal push notifications

Push scaffold for secure messaging behind `ENABLE_PORTAL_PUSH` + `ENABLE_CLIENT_PORTAL`:

| Endpoint                                 | Description                                                  |
| ---------------------------------------- | ------------------------------------------------------------ |
| `GET /portal/push/status`                | Readiness, blockers, subscription count                      |
| `POST /portal/push/subscribe`            | Register Web Push subscription keys                          |
| `DELETE /portal/push/subscriptions/{id}` | Deactivate subscription                                      |
| Dispatch                                 | Staff replies on linked cases enqueue push delivery attempts |
| UI                                       | `PortalPushPanel` on portal case detail                      |

Web Push provider is scaffold-only (audit logs, no external push HTTP yet). Real Web Push delivery → **5.2+**.

## Materialized reporting

Read-optimized bureau and team aggregates behind `ENABLE_MATERIALIZED_REPORTING`:

| Capability | Details                                                                                           |
| ---------- | ------------------------------------------------------------------------------------------------- |
| Views      | `mv_bureau_account_counts`, `mv_bureau_sent_letter_counts`, `mv_team_member_productivity`         |
| Read paths | `GET /reporting/bureau-performance`, `GET /reporting/team-productivity` use MVs when flag enabled |
| Refresh    | `POST /reporting/materialized-views/refresh` (admin)                                              |
| Audit      | `GET /reporting/materialized-views/status`, `GET …/runs`                                          |
| Worker     | `reporting_mv_refresh` at `0 4 * * *` UTC                                                         |

Revenue metrics and score-improvement trends → **5.2+**.
