# Release Notes — Version 4.8.0 (Operations)

**Version:** 4.8.0  
**Date:** 2026-07-02  
**Tag:** `v4.8.0`

## Summary

Version 4.8.0 is the **Operations** release. It extends the Automation Platform (v4.5.0) with staff notifications, scheduled workflow jobs, client management, a client portal, LLM policy gates, email delivery scaffolding, and operational reporting — all behind feature flags and documented **Partial** scope limits.

Five epics ship as **Partial** with explicit deferrals to 5.0 for enterprise compliance, production email/SMS, LLM summary endpoints, and full job orchestration parity. See [`docs/governance/version-4.8-scope.md`](../governance/version-4.8-scope.md).

## Shipped in v4.8.0

| Area                    | Highlights                                                                                     |

| **Notifications**       | Persisted in-app notifications, unread count, mark read, staff notification bell UI            |
| **Email scaffold**      | `ENABLE_EMAIL_DELIVERY` + `GET /notifications/email/status` readiness gate (no provider sends) |

| **Reporting**           | `GET /reporting/operations` read model; `operations` section on Mission Control dashboard      |
| **Governance**          | Capability matrix 4.8 sign-off, scope doc, completion checklist                                |

## Notifications

Staff users receive persisted in-app notifications:

- `POST/GET /notifications`, unread count, mark one/all read
- Notification bell in staff web app
- Admin-only create endpoint; modules should call `NotificationService.notify_user()` for workflow events (follow-up wiring)

Email delivery is scaffolded only — configure `ENABLE_EMAIL_DELIVERY` and provider env vars; use `/notifications/email/status` to verify readiness without sending mail.

## Workflow & job orchestration

- **`overdue_investigation_scan`** worker replaces the 4.5 read-time escalation on account GET
- **`packages/job-orchestrator/`** (`verdin-job-orchestrator`) — shared job contracts, registry, Redis queue, retry/scheduler/metrics scaffolds
- Runner retry wiring, PostgreSQL job persistence, and cron UI remain deferred

## Client experience

| Capability             | Endpoints / UI                                     |
| ---------------------- | -------------------------------------------------- |
| Client + contacts      | `/clients`, `/clients/{id}/contacts`               |
| Portal auth            | `/portal/auth/login`, `/refresh`, `/me`            |
| Portal provisioning    | `/clients/{id}/portal-user` (staff, case_manager+) |
| Case progress (portal) | `/portal/cases`, `/portal/cases/{id}`              |
| Portal web UI          | `/portal/login`, `/portal`, `/portal/cases/:id`    |

Portal case matching uses email/name heuristics until optional `cases.client_id` linking ships.

Enable with `ENABLE_CLIENT_PORTAL=true` (API) and `VITE_ENABLE_CLIENT_PORTAL=true` (web).

## LLM policy gates

No external LLM provider calls ship in v4.8.0:

- [`docs/adr/012-llm-provider-policy.md`](../adr/012-llm-provider-policy.md)
- `packages/llm-gateway/` — PII scrubbing, `require_llm_ready()`, provider config
- `ENABLE_LLM` separate from heuristic `ENABLE_AI`
- `GET /llm/status` for staff readiness checks

Case/document summaries and LLM dispute augmentation → **5.0** (post-gate implementation).

## Operations reporting

- `GET /reporting/operations` — org-scoped aggregates for clients, dispute accounts/letters, notifications
- Mission Control `GET /dashboard` embeds the same `operations` read model
- Materialized views / read replicas deferred if latency targets are exceeded
