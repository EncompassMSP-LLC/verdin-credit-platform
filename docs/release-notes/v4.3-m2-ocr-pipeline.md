# Version 4.3 — Document Intelligence Milestone 2: OCR Pipeline

**Release tag:** `v4.3.0-m2`  
**Branch:** `feature/document-ocr` (merged to `main`)

## Added

### Backend (`apps/api`)

- Migration `005_document_ocr` — `processing_status`, `ocr_text`, `ocr_error`, `ocr_processed_at`, `ocr_job_id`, `ocr_version_number`
- Redis job enqueue (`api/core/job_queue.py`) after document upload and version upload
- OCR endpoints: `GET /documents/{id}/ocr`, `POST /documents/{id}/ocr/retry`
- List filter: `processing_status`
- Eligible formats (PDF, JPG, PNG, TIFF) enqueue async OCR; DOCX/TXT marked `skipped`

### Worker (`apps/worker`)

- `OcrJob` — reads MinIO, extracts text via pypdf (PDF) and pytesseract (images)
- Persists OCR results to PostgreSQL with processing status lifecycle
- Tesseract installed in worker Docker image for image OCR

### Frontend (`apps/web`)

- Processing status badges on document list and detail
- Extracted text panel with auto-refresh while queued/processing
- Retry OCR action on failure
- OCR status filter on document library

### Packages

- `@verdin/shared` — `DocumentProcessingStatus` types and labels
- `@verdin/api-client` — `getDocumentOcr`, `retryDocumentOcr`

## Infrastructure

- Test harness: per-request DB sessions (fixes asyncpg cross-loop errors in CI)
- Auth: direct `bcrypt` (replaces unmaintained passlib); passwords over 72 UTF-8 bytes rejected at validation

## Pipeline

```
Upload → OCR queue (Redis) → Worker → Extract text → Store in PostgreSQL
```

Upload requests return immediately; OCR never blocks HTTP.
