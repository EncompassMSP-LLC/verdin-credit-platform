# Verdin Credit Platform — Version 5.0 Enterprise Roadmap

**Status:** Active master roadmap  
**Last updated:** 2026-07-02

## Vision

Build the industry's most comprehensive AI-powered Credit Repair Operations Platform.

Version 5.0 transforms Verdin from a case management application into a complete enterprise platform supporting credit repair organizations, legal teams, financial consultants, and compliance-focused operations.

Every sprint between now and 5.0 should move a measurable capability forward while preserving the platform principles and engineering standards defined below.

---

## Platform principles

| Principle                     | Meaning for Verdin                                                                                                      |
| ----------------------------- | ----------------------------------------------------------------------------------------------------------------------- |
| **AI-first architecture**     | AI capabilities are designed into domains (not bolted on); every data object should be consumable by models and agents. |
| **API-first design**          | All product features exist as versioned, documented APIs before UI-only shortcuts.                                      |
| **Multi-tenant by default**   | Organization scoping, RBAC, and data isolation are non-negotiable from day one.                                         |
| **Event-driven architecture** | Domain actions emit events for timeline, automation, analytics, and audit.                                              |
| **Secure by design**          | Authentication, authorization, encryption, and audit are built into every module.                                       |
| **Compliance-first**          | CROA, FCRA, FDCPA, consent, and retention requirements inform data models and workflows.                                |
| **Cloud-native deployment**   | Stateless services, horizontal scaling, containerized workloads.                                                        |
| **Modular services**          | Domain modules with clear boundaries (router → service → repository).                                                   |
| **Enterprise scalability**    | Performance, observability, and operational tooling suitable for high-volume shops.                                     |

## Core platform domains

### Client Management

Manage organizations, clients, contacts, and onboarding.

**Modules:** Organizations, Clients, Contacts, Intake, Client Portal

**Current state (4.8):** Client/contact CRUD, portal auth, read-only case progress — **shipped in v4.8.0**. Upload, messaging, billing — **5.0**.

---

### Case Management

The central object within the platform.

**Capabilities:**

- Case lifecycle
- Assignments
- Status tracking
- Priority management
- Evidence collection
- Internal notes
- AI summaries

**Current state (4.3):** Full CRUD, filters, RBAC, frontend — **Sprint 2 Epic 1 shipped**.

---

### Credit Account Intelligence Engine

Enterprise account intelligence for tradelines and dispute readiness.

**Capabilities:**

- Credit account management
- Bureau tracking
- Risk scoring
- Readiness scoring
- Dispute readiness
- AI recommendations
- Historical comparisons

**Current state (4.3):** CRUD, intelligence summary, scoring heuristics, case-linked accounts — **Sprint 2 Epic 2 shipped**. Historical comparisons and advanced AI recommendations are planned for 4.5+.

---

### Document Intelligence Platform

**Capabilities:**

- Secure storage
- OCR
- AI classification
- Metadata extraction
- Entity recognition
- Version history
- Search
- Duplicate detection

**Target version:** 4.3 (foundation), 4.5 (OCR + classification)

---

### Timeline & Audit Engine

Every system action creates an immutable audit event.

**Examples:** Case created, account imported, document uploaded, OCR completed, letter generated, task assigned, user login, client notification

**Target version:** 4.3 (event model + API), 5.0 (immutable enterprise audit store)

---

### Workflow Automation

Automate operational processes.

**Examples:** Assign reviewers, schedule disputes, generate reminders, trigger AI reviews, follow-up workflows, SLA monitoring

**Target version:** 4.5

---

### Task Management

Operational work management.

**Features:** Due dates, priorities, queues, dependencies, automation, escalation

**Current state:** Model and scaffold exist; full module parity with cases/accounts planned for 4.3.

---

### Credit Report Import Engine

**Supports:** PDF import, OCR, native parsing, bureau detection, duplicate detection, snapshot comparison, historical tracking

**Target version:** 4.5

---

### Dispute Generation Engine

**Generate:** Bureau disputes, furnisher disputes, CFPB complaints, FTC documentation, attorney correspondence

AI-assisted generation with reusable templates.

**Target version:** 4.5 (templates + generation), 5.0 (autonomous preparation)

---

### AI Case Assistant

**Provides:** Case summaries, recommended actions, missing evidence, next dispute strategy, bureau response analysis, consumer coaching

**Target version:** 4.5 (Phase 2 summaries), 5.0 (Phase 3–4 orchestration)

---

### Reporting & Analytics

Operational dashboards: open cases, resolution rates, bureau performance, average score improvement, team productivity, revenue metrics, compliance reporting

**Target version:** 4.8

---

### Client Portal

Secure portal for clients: upload documents, view case progress, secure messaging, tasks, notifications, credit score history

**Target version:** 4.8

---

### Communications Center

Unified communications: email, SMS, internal notes, client messaging, letter history, notifications

**Target version:** 4.8–5.0

---

### Compliance Center

Track CROA, FCRA, FDCPA, CFPB, audit logs, consent history, data retention

**Target version:** 5.0

---

### Enterprise Administration

Organization management: multi-tenancy, RBAC, SSO, API keys, billing, feature flags, audit management

**Current state:** RBAC and feature flags (foundation). SSO, billing, API keys — **5.0**.

## Engineering standards

Every feature must include:

- Documentation (API reference + architecture alignment)
- Tests (unit + integration for endpoints)
- RBAC enforcement
- Audit / timeline events (where applicable)
- API documentation (OpenAPI + `docs/api/reference.md`)
- Frontend integration (`@verdin/api-client`, `@verdin/ui`)
- Mobile responsiveness
- Accessibility baseline
- CI/CD validation (pre-commit, GitHub Actions)
