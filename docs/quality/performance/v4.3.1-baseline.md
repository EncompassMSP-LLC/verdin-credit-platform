# v4.3.1 Performance Baseline

**Purpose:** establish reproducible Sprint 4.3.1 performance baselines before Version 4.5 automation work begins.

The goal is not to optimize prematurely or hit every target immediately. The goal is to capture a stable reference point so future changes can be compared against known Operational Core behavior.

## Measurement Rules

- Use the golden-path E2E fixture shape where possible: one organization, one owner user, one case, one account, one 1-page credit-report PDF, one task, and Mission Control reads scoped to the test organization.
- Record the environment for every run: local machine or CI, OS, Python/Node versions, database location, worker count, and whether services are containerized.
- Run each synchronous endpoint at least 10 times after one warm-up request; record median, p95, and max where practical.
- Measure asynchronous worker stages from job enqueue to terminal state using timestamps or structured worker logs.
- Keep raw notes or scripts alongside future updates if measurement automation is added.

## Results Notes

- Dashboard API: well under the initial `<500 ms` target in the local stack.
- Login: above the initial `<150 ms` target. Do not optimize by guesswork; profile the login path first.
- Create Case: well under the initial `<200 ms` target.
- Upload Document: under the initial `<500 ms` target excluding downstream OCR.
- OCR: 1-page synthetic PDF averaged `182.43 ms`; 3-page synthetic PDF averaged `140.61 ms`. The 3-page run was faster in this sample due to small fixture size and local variance; rerun in CI before drawing scaling conclusions.
- Classification: average `89.82 ms`; p95 `228.94 ms`.
- Metadata Extraction: average `76.22 ms`; p95 `193.84 ms`.
- Entity Resolution: average `72.87 ms`; p95 `130.36 ms`.
- Timeline Write: well under the initial `<100 ms` target.

## Login Profiling Follow-Up

Before changing authentication behavior, break `POST /api/v1/auth/login` into measured stages:

1. Request validation and schema parsing.
2. User lookup by email.
3. Password verification (`bcrypt`).
4. Role and organization loading.
5. Access token generation.
6. Refresh token generation.
7. Response serialization.

Expected hypothesis: password verification is the dominant contributor. Treat this as a hypothesis until measured.

## Candidate Performance Regression Gate

Do not make this gate blocking until the baseline is re-measured in CI and thresholds are calibrated against CI variance. Initial candidate thresholds:

| Metric          | Candidate CI Failure Threshold | Rationale                                  |
| --------------- | ------------------------------ | ------------------------------------------ |
| Dashboard API   | `>250 ms avg`                  | Catches large regressions while below SLA. |
| Create Case     | `>100 ms avg`                  | More than 4x local baseline.               |
| Timeline Write  | `>25 ms avg`                   | More than 3x local baseline.               |
| Document Upload | `>300 ms avg` excluding OCR    | More than 2x local baseline.               |

Recommended rollout:

1. Add a non-blocking CI benchmark job that uploads raw JSON output from [`measure_v431_baseline.py`](measure_v431_baseline.py).
2. Run it across several merges to establish CI variance.
3. Enable blocking thresholds only for stable, low-variance metrics.
4. Keep asynchronous worker pipeline timings as reported baselines first; gate them only after CI variance is understood.
